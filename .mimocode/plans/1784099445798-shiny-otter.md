# Implementation Plan: WinPaste Feature Parity

## Overview

This plan closes the feature gap between the current Clipboard project (WinUI3 native) and WinPaste (Tauri+React). The database schema is already rich (10 tables, 40+ settings), and many Win32 APIs are wrapped. The main gaps are: **settings UI disconnected from DB**, **simplified paste pipeline**, **missing keyboard navigation**, **incomplete tag system UI**, and **missing content type handling**.

Work is organized into 8 phases, each independently shippable. Within each phase, tasks are ordered by dependency.

---

## Phase 1: Fix Settings UI <-> DB Connection

**Goal**: Make every settings toggle actually persist to and read from the database.

### Files to modify
- `src/ui/settings_window.rs` — Replace hardcoded `SettingsState::default()` with real DB reads
- `src/app/setup.rs` — Ensure `SettingsState` is populated from DB on startup
- `src/state/app_state.rs` — Add missing fields if needed

### Tasks
1. **Load settings from DB into `SettingsState` on startup** — In `app::setup::init()`, after DB init, read all `app.*` keys from `SettingsRepository` and populate `SettingsState` atomics/mutexes. Currently `SettingsState::default()` returns hardcoded values.
2. **Wire each settings toggle to its DB key** — Each `setting_toggle()` call in `settings_window.rs` already calls `save_bool_setting()`, but the initial value comes from `SettingsState::default()`. Change to read from the loaded state.
3. **Fix mismatched setting keys** — Several toggles in `settings_window.rs` are wired to the wrong keys (e.g., "贴图功能" uses `capture_files` instead of `sticky_enabled`, "标签管理页开关" uses `sequential_mode` instead of `tag_manager_enabled`). Map each UI toggle to its correct DB key.
4. **Add missing settings UI rows** — The current settings panel has ~24 rows; WinPaste has ~40. Add rows for: `follow_caret`/`follow_mouse` position picker, `edge_docking`, `sound_volume` slider, `paste_sound`, `rich_paste_hotkey` recording, `search_hotkey` recording, `sequential_hotkey` recording, `arrow_key_selection`, `auto_hide_tags`, `pinned_collapsed`, `show_source_app_icon`, `show_app_border`, `vibrancy_enabled`.

## Phase 2: Fix Paste Pipeline

**Goal**: Correctly paste images (CF_DIB), files (CF_HDROP), and rich text (HTML) instead of writing base64 as text.

### Files to modify
- `src/services/clipboard_ops.rs` — Update `paste_entry()` to call correct Win32 clipboard write APIs per content type
- `src/infrastructure/windows_api/win_clipboard.rs` — Already has `set_clipboard_image_and_gif()`, `set_clipboard_text_and_html()`, `set_clipboard_files()`

### Tasks
1. **Image paste** — In `paste_entry()`, when `content_type == "image"`, decode the image content (file path or data URL) to bytes, then call `set_clipboard_image_with_formats()` instead of `set_clipboard_text()`.
2. **File paste** — When `content_type == "file"`, parse newline-separated paths and call `set_clipboard_files()`.
3. **Rich text paste** — When `content_type == "rich_text"` and `html_content` is `Some`, call `set_clipboard_text_and_html()` with both text and HTML.
4. **Sequential paste mode** — When `sequential_mode` is enabled, instead of pasting immediately, enqueue the entry ID into `PasteQueue` and send the next item on each hotkey press. Wire the `sequential_hotkey` (Alt+V) to dequeue + paste.
5. **Delete after paste / move to top** — After successful paste, if `delete_after_paste` is true, delete the entry from DB and remove from cache. If `move_to_top_after_paste` is true, bump `use_count` and reorder.

## Phase 3: Keyboard Navigation

**Goal**: Arrow keys navigate the list, Enter pastes, Escape hides.

### Files to modify
- `src/ui/main_window.rs` — Add keyboard event handling
- `src/state/global_state.rs` — Already has `NAVIGATION_ENABLED`, `NAVIGATION_MODE_ACTIVE`

### Tasks
1. **Track selected index** — Add `SELECTED_INDEX: AtomicI32` to global state. Initialize to 0 when window shows.
2. **Wire key events in main_window** — In the WinUI3 event loop, intercept `VirtualKey::Up`/`Down` to move selection, `VirtualKey::Enter` to paste selected, `VirtualKey::Escape` to hide, letter keys to jump-search.
3. **Visual selection highlight** — In `clipboard_item()`, check if `entry index == SELECTED_INDEX` and apply a different background color (e.g., accent color with low opacity).
4. **Auto-scroll to selected** — When selection changes, scroll the `ScrollViewer` to keep the selected item visible.

## Phase 4: Content Type Enhancements

**Goal**: Support video thumbnails, file system icons, source app icons.

### Files to modify
- `src/ui/content_preview.rs` — Add video card renderer, enhance file card with system icon
- `src/infrastructure/windows_api/apps.rs` — Implement `get_system_default_app()`
- `src/infrastructure/windows_api/win_clipboard.rs` — Ensure image capture works end-to-end

### Tasks
1. **Video content type** — Add `render_video_card()` in `content_preview.rs` that shows a video icon + filename. The `ClipboardEntry` model already has `content_type: "video"`.
2. **File type icons** — For file cards, extract the file extension and use appropriate emoji/icon (`.zip`→📦, `.mp3`→🎵, `.pdf`→📄, `.exe`→⚙️). This matches WinPaste's `getFallbackFileIcon()`.
3. **Source app color** — The existing `app_icon_color()` in `main_window.rs` is already decent. Keep it but also add it to the `render_card()` dispatch in `content_preview.rs` so all card types show the source indicator.
4. **Rich text preview** — Add `render_rich_text_card()` that shows the HTML content in a simplified text form (strip tags, show first 160 chars of text content). Full HTML rendering in WinUI3 Webview would be a future enhancement.
5. **URL type detection** — In the clipboard monitor, detect if text content is a URL (starts with `http://` or `https://`) and set `content_type = "url"` instead of `"text"`.

## Phase 5: Tag System UI

**Goal**: Working tag management: assign tags to entries, filter by tag, tag manager page.

### Files to modify
- `src/ui/main_window.rs` — Tag filter in search bar, tag chips on items
- `src/ui/tag_manager.rs` — **NEW FILE** — Full tag manager page
- `src/infrastructure/repository/tag_repo.rs` — Already functional

### Tasks
1. **Tag chips on clipboard items** — In `clipboard_item()`, render small colored tag chips below the content preview. Read `entry.tags` and render each with `saved_tags` color.
2. **Tag filter dropdown** — In the search bar area, when search is empty and focused, show a dropdown of all tags from `TagRepository::get_all_tags()`. Clicking a tag sets search to `tag:tagname`.
3. **Tag editor on items** — Add a tag icon button on each item. Clicking it opens an inline tag input where you can type a tag name and press Enter to add. The `TagRepository` already has `add_tag_to_entry()`.
4. **Tag manager page** — Create `src/ui/tag_manager.rs` with a left sidebar (tag list with counts, rename, delete, color picker) and right content area (items for selected tag). Wire the Tag button in the command bar to toggle this view.
5. **Search tag:xxx syntax** — In the search filter logic, detect `tag:xxx` prefix and filter entries by tag using `TagRepository::get_entries_by_tag()`.

## Phase 6: Privacy Protection

**Goal**: Detect and mask sensitive content (phone, ID, email, password).

### Files to modify
- `src/infrastructure/repository/clipboard_repo.rs` — Add privacy scan during insert
- `src/ui/main_window.rs` — Show masked content with reveal button

### Tasks
1. **Privacy scanner function** — Create `src/services/privacy.rs` with regex-based detection for:
   - Phone: `1[3-9]\d{9}`
   - ID card: `\d{17}[\dXx]`
   - Email: standard pattern
   - Password: 8+ chars with upper+lower+digit+special
   - Custom rules: user-provided regex from `privacy_protection_custom_rules` setting
2. **Auto-tag sensitive entries** — In the clipboard monitor pipeline, after creating a `ClipboardEntry`, run the privacy scanner. If matched, add the `"sensitive"` tag (already seeded in DB) via `TagRepository`.
3. **Mask in UI** — In `clipboard_item()`, if entry has `"sensitive"` tag and `privacy_protection` setting is on, show blurred/masked content with an Eye/EyeOff toggle button. Store revealed IDs in a `HashSet<i64>` in component state.
4. **Kind-based filtering** — Read `privacy_protection_kinds` setting to control which detection types are active.

## Phase 7: Window Polish

**Goal**: Follow-caret/mouse positioning, edge docking, window pin, compact mode improvements.

### Files to modify
- `src/app/window_manager.rs` — Add caret/mouse position detection
- `src/ui/main_window.rs` — Compact mode layout
- `src/ui/styles.rs` — Add compact mode dimensions

### Tasks
1. **Follow caret position** — When `follow_caret` setting is on, use UIAutomation (`IUIAutomation::GetFocusedElement` → `GetBoundingRectangle`) to get the text caret position and show the window near it. The `window_tracker.rs` already imports `windows::Win32::UI::Accessibility`.
2. **Follow mouse position** — When `follow_mouse` is on (default), use `GetCursorPos()` to position window near the mouse. Already partially implemented.
3. **Screen center** — When neither follow mode is on, center the window on the primary monitor.
4. **Compact mode** — When `IS_COMPACT` is true, reduce item height, hide metadata, show only preview text + source dot. Update `clipboard_item()` to check the compact flag and render a minimal layout.
5. **Window pin** — The `toggle_pin()` function already exists. Wire the Pin button in the title bar command bar to call it and toggle `WINDOW_PINNED`.

## Phase 8: Data Management & Extras

**Goal**: Clear history, data path, version display, sound effects, hotkey recording.

### Files to modify
- `src/ui/settings_window.rs` — Clear history button, data path display
- `src/main.rs` — Version display
- `src/services/hotkey.rs` — Support configurable hotkeys

### Tasks
1. **Clear history** — Wire the "清空历史" button to call `ClipboardRepository::delete_all()` (add this method) and clear `CLIPBOARD_ENTRIES`.
2. **Version display** — Read version from `Cargo.toml` at compile time via `env!("CARGO_PKG_VERSION")` and display in settings footer.
3. **Hotkey recording UI** — Add a custom hotkey recording widget: show current hotkey as key caps, clicking enters recording mode, next keypress saves new hotkey. Wire to `parse_hotkey_string()` and re-register via `RegisterHotKey`.
4. **Sound effects** — When `sound_enabled` is on, play a system sound on copy/paste events. Use `MessageBeep(MB_OK)` or play a custom WAV from resources.
5. **Default apps** — Implement `get_system_default_app()` in `apps.rs` using `SHQueryAssociations` or `AssocQueryString` to find the default handler for a file type. Display in settings.

## Verification

After each phase:
1. `cargo check` — Ensure compilation succeeds
2. `cargo test` — Run existing tests
3. Manual test: launch with `cargo run`, exercise the new feature, verify settings persist across restart

After all phases:
- Compare feature-by-feature with WinPaste to confirm parity
- Test paste pipeline with text, images, files, and rich text
- Test keyboard navigation with arrow keys
- Test tag CRUD and filtering
- Test privacy detection with sample sensitive data
- Test hotkey recording and custom hotkeys
